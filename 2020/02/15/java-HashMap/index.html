HashMap

这个2020春节，我接触到了慕课网(imooc.com)这个程序员提升的平台,发现了很多有意思而且有内容的课程.其中之一是<剖析HashMap源码>, 学习完之后不禁地就问自己,能否整理一些自己对HashMap的理解, 也许下一次自己给朋友分享的时候会比慕课的老师一样好。

HashMap 与 ArrayList是Java代码中非常高频出现的集合类. 本文将会介绍HashMap的功能,分析Java JDK1.8的源代码.

HashMap的基础核心架构是一个动态数组,当不同的key经过hash散列函数后,得到相同的散列值,数组位置存放的是一个链表. 简而言之, 动态数组+单链表的复合结构, 是HashMap的组织形式.

在JDK1.6, HashMap的结构如上图所示,但我们知道单链表的随机遍历的复杂度是O(n), 如果单链表长度较长(即Hash冲突严重)时,HashMap的性能也会相应降低.

在JDK1.8中, 对此进行了优化,即用著名的红黑树(Red-Black Tree)处理这种情况,后方文中也会对红黑树的实现细节进行讲解.

hash 方法

对于HashMap来说, 除了其数据结构的构造外, 我们还忽略了其打散 key值的核心的散列函数 hash 函数, 它在JDK8也做了一些修改, 相较于之前的版本(四次位运算),而简化成为2次: 先取出key自己的hashCode记为h, 然后和右移16位后的h进行异或位运算.

总结： hash函数中强制指定了要高位结果也参与运算（右移16位后剩余的即为高位），目的是为了散列冲突。 并且出于效率的考虑，在Java8中这种位移动做了次数精简。

分析: 我们知道在Java中所有Object对象都拥有一个叫做hashCode 的方法,理论上它会返回一个int数字(范围是 -2^31 ~~ 2^31) 这是一个近43亿的长度——不过显然, 我们的HashMap初始化不会一上来就开辟这么大的空间，初始大小只有16.

而且在这里，能进一步看到另一个结论：HashMap的数组长为什么要设计成2的整数次幂？ 在对key的操作还出现了 tab[(n - 1) & hash]) 这个操作是对n-1， 这样得到了一个类似于 0000 0000 1111 1111 1111 1111 这种低位全是1， 高位全是0的掩码（mask）， 与hash值进行与运算时只会保留hash的低位。

线程安全问题

在多线程环境下,使用HashMap的put操作会存在线程安全问题, 主要原因是 put的密集操作会引起HashMap容量上限的扩增,List链表有可能变成首尾有环的循环链表,导致CPU利用率达100% ,所以在并发环境下,不建议使用HashMap.

    • 《Java 并发编程的艺术》 — 方腾飞等

这里我在 stack overflow 上找到了这篇答案 ， 个人觉得解释的更好一些。

主要原因：

当多个线程快速向 HashMap 添加元素时，HashMap#put() 方法会调用 HashMap#resize() 扩容，这个过程是内存要开辟一个 新数组，然后进行数据迁移及潜在的 treeify 过程。如果此时另一线程调用了 HashMap#get()方法，试图取出元素，但恰恰发生数据迁移时，就有可能产生 元素不存在的异常。
